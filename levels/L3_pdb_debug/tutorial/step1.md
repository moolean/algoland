# Step 1/3 - 教学：用 PDB 看 JSONL 打分脚本为什么在边界样本上算错了

## 任务速览
- 录制一份 PDB 调试会话到 `out/pdb_session.txt`
- 调试 `lab/pdb_bug.py`
- 查看筛选后的样本和分母使用是否合理

## 这一步你在学什么
PDB 的核心不是背命令，
而是：**让程序停在出问题的位置，直接看现场。**

这一步的代码更像算法离线分析：
- 从 JSONL 读样本
- 按条件筛选样本
- 聚合 score
- 生成最终打分

你要学的是：
- 怎么把脚本放进调试器里运行
- 怎么让它跑到关键位置
- 怎么看筛选结果和聚合逻辑到底哪里不对

## 核心概念
### 1) `python -m pdb lab/pdb_bug.py`
这条命令表示：
- 不按普通方式运行脚本
- 而是用 PDB 接管执行过程

进入后你会看到 `(Pdb)`，说明程序停住了，正在等调试命令。

### 2) `c` 为什么会直接跑完
`c` 的意思是 continue。
它会让程序一直往下跑，直到：
- 遇到断点
- 遇到异常
- 或者直接运行结束

所以如果你什么断点都没下，
`c` 很可能就会一路跑到结尾。

也就是说，这题里如果你想看 `selected`，
不能只靠 `c`，要先让程序**停在 `build_score()` 里面**。

### 3) 这题里更合适的调试命令
这一步最常用的是：
- `l`：看当前代码
- `b build_score`：在 `build_score()` 入口下断点
- `c`：继续运行，直到断点或结束
- `n`：执行下一行，但不进入别的函数
- `p selected`：看筛选后的样本
- `p len(selected)`：看筛选后的样本数
- `q`：退出调试器

你可以先把它们理解成：
- `b` 负责“先拦住程序”
- `c` 负责“让程序跑到你拦下的位置”
- `n` 负责“在当前函数里一行一行往下走”

### 4) 怎么把调试过程录到文件
这题要求把整段 PDB 会话录下来。

在这台机器上，推荐直接用 `script` 包住调试命令：

```bash
script -q -F -k out/pdb_session.txt python -m pdb lab/pdb_bug.py
```

### 4) 这题的问题在哪
脚本会先筛出 `keep=true` 的样本，再做平均分。

但当前实现里，
分子用的是筛选后的 score 之和，
分母却还在用原始样本总数。

这类 bug 很像真实工作里的离线分析错误：
**样本筛选和聚合口径不一致。**

## 你在这一步怎么走
推荐顺序：
1. 运行：
```bash
script -q -F -k out/pdb_session.txt python -m pdb lab/pdb_bug.py
```
2. 输入 `l`，先看当前停在哪
3. 输入 `b build_score`，先在 `build_score()` 入口下断点
4. 输入 `c`，让程序继续跑到这个断点
5. 进入 `build_score()` 后，输入 `n` 往下执行到 `selected = ...` 这一行后面
6. 输入 `p selected`
7. 输入 `p len(selected)`
8. 输入 `q`
9. 如果又回到脚本开头，再输入一次 `q`
